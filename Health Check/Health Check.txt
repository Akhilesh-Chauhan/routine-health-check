## How to Check Each URL
For each URL, use your available tools (bash, curl, or HTTP fetch) to:
1. Send an HTTP GET request with a 10-second timeout
2. Record the HTTP status code
3. Record the response time in milliseconds
4. Note any connection errors (DNS failure, SSL error, timeout, refused)

Use this classification:
- **UP** → HTTP 2xx or 3xx, response under 4000ms
- **SLOW** → HTTP 2xx or 3xx, response over 4000ms
- **DOWN** → HTTP 4xx/5xx, timeout, DNS failure, or connection refused

Check all URLs in parallel batches of 5 to be efficient. Do not stop on a failure — check every single URL regardless.

## Output Format
After checking all URLs, produce this exact report structure:

### 1. Executive Summary
- Total checked / Online / Slow / Down
- Overall system health: HEALTHY / DEGRADED / CRITICAL
- Timestamp (IST)

### 2. Incidents (if any)
For each DOWN or SLOW service, provide:
- Service name and URL
- Environment (Production / Dev / Chatbot)
- Failure reason (exact error or HTTP code)
- Response time
- Severity: CRITICAL (prod down) / HIGH (prod slow) / MEDIUM (dev/chatbot down) / LOW (dev slow)
- Recommended action

### 3. Full Results Table
A table with columns: Label | URL | Status | HTTP Code | Response (ms) | Error

### 4. Environment Breakdown
Summarize UP/SLOW/DOWN counts separately for Production, Dev, and Chatbot groups.

### 5. Recommendations
Based on the results, suggest any patterns, recurring failures, or infrastructure concerns worth escalating.

## Constraints
- Never skip a URL — report every one, even if it times out
- Do not assume a service is up without actually checking it
- If a tool call fails, retry once before marking the URL as DOWN
- Be precise with error messages — "Connection refused" is more useful than "failed"
- All timestamps must be in IST (UTC+5:30)
